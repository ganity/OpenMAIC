## 1. 设计目标与原则

### 1.0 阅读说明

本章节定位为**目标设计文档**，不是当前实现说明文档。

建议按三层理解全文：

1. **当前现状**：系统当前真实已具备的能力与约束
2. **目标设计**：企业培训模板体系的目标形态
3. **实施路径**：从当前状态演进到目标状态的执行计划

当前代码现状需要特别说明：

- 当前 `UserRequirements` 仍以 `requirement` 自由文本为主
- 当前系统还没有原生的 `trainingType`、`sourceMode`、`riskLevel`、`templateFamily` 等结构化字段
- 文档中这些字段与机制，主要属于目标设计与实施方向，不应被理解为当前系统已全部具备

### 1.1 设计目标

由于产品面向企业市场，实际培训主题无法完全预判，因此不建议预制“具体内容模板”，而建议预制“生成策略模板 / 课件骨架模板”。

模板体系的目标不是提前写死某个培训主题，而是让系统具备：

1. 识别培训类型的能力
2. 根据培训类型切换生成策略的能力
3. 在有企业资料时严格受资料约束的能力
4. 在无企业资料时输出通用模板与占位内容的能力
5. 允许企业沉淀自己的模板资产的能力

### 1.2 设计原则

企业培训模板体系建议遵循以下原则：

- 少量模板族覆盖多数场景，而不是无限细分模板
- 模板作用于“生成逻辑”，不作用于“固定内容”
- 模板选择要结合培训类型、资料模式、风险等级、交付方式
- 制度/合规类模板必须具备强约束，禁止编造制度细节
- 技能类模板必须强调业务场景、练习、误区与岗位迁移
- 支持企业自定义模板覆盖系统默认模板

### 1.3 三层模板结构

#### 第一层：通用元模板

该层不承载具体培训内容，而承载所有模板共用的输入约束字段。

建议字段：

- `trainingType`：培训类型
- `sourceMode`：资料模式
- `audience`：培训对象
- `duration`：时长
- `deliveryMode`：线上/线下/混合
- `assessmentNeeded`：是否需要考核
- `mustInclude`：必须包含内容
- `forbidden`：禁止生成内容
- `tone`：风格要求
- `riskLevel`：风险等级

#### 第二层：系统模板族

系统内置少量模板族，用于覆盖绝大多数企业培训课件生产场景。

建议首批模板族：

1. 入职 / 融入模板
2. 制度 / 合规模板
3. 流程 / SOP 模板
4. 系统操作模板
5. 技能模板
6. 产品 / 服务知识模板
7. 安全 / 应急模板

#### 第三层：企业自定义模板

企业管理员可以在系统模板族基础上沉淀自己的模板，例如：

- 企业新员工入职模板
- 企业制度培训模板
- 门店培训模板
- 区域销售培训模板

企业自定义模板可以覆盖：

- 固定首页结构
- 固定模块顺序
- 固定讲师备注格式
- 固定测验题型
- 固定品牌/企业话术
- 固定页风格与内容边界

## 2. 七大模板族设计

### 2.1 入职 / 融入模板

#### 适用场景

- 新员工入职培训
- 企业文化导入
- 组织架构介绍
- 基础制度总览
- 岗位认知入门

#### 模板目标

帮助新员工快速理解“公司是什么、我需要知道什么、哪些规则最重要、日常工作怎么开展”。

#### 推荐输出结构

1. 培训目标
2. 适用对象
3. 企业/团队/岗位基本认知
4. 新员工必须知道的关键制度
5. 高频工作场景说明
6. 常见问题与误区
7. 风险提醒
8. 小测验 / 确认环节

#### Prompt 约束重点

- 避免过度宣传式内容堆砌
- 强调新人视角友好
- 信息分层，避免一页过载
- 关键制度点要配场景说明
- 没有企业资料时，必须输出“需企业补充”占位项

### 2.2 制度 / 合规模板

#### 适用场景

- 考勤制度
- 奖惩制度
- 保密制度
- 行为规范
- 安全管理制度说明
- 合规要求

#### 模板目标

把制度内容转化为可培训、可理解、可执行的课件，同时保持制度含义不失真。

#### 推荐输出结构

1. 适用对象与适用范围
2. 核心规则点
3. 适用边界与例外情况
4. 高频误区
5. 风险与后果
6. 场景判断题 / 判断案例
7. 培训确认与答疑

#### Prompt 约束重点

- 如果 `sourceMode = strict_grounded`，只允许基于企业资料生成
- 不得虚构公司具体制度条款
- 对不明确内容必须标注“需企业确认”
- 优先输出规则、边界、例外、后果
- 语言应准确、克制、清晰

#### 归类边界说明

当培训重点是“规则说明、适用范围、禁止事项、责任与后果”时，应优先归到制度 / 合规模板。

如果主题同时包含制度与应急，应优先按**主要训练目标**归类，而不是按字面名词归类。


### 2.3 流程 / SOP 模板

#### 适用场景

- 报销流程培训
- 请假流程培训
- 客诉处理流程培训
- 生产作业流程培训
- 审批流与协同流程培训

#### 模板目标

把企业流程拆解为清晰、可执行、可复现的步骤型课件，帮助学员知道“什么时候做、怎么做、做错会怎样、异常情况怎么处理”。

#### 推荐输出结构

1. 流程适用场景
2. 流程目标与完成标准
3. 标准步骤拆解
4. 关键节点与责任角色
5. 异常情况处理
6. 高频错误与返工点
7. 流程演练 / 场景题

#### Prompt 约束重点

- 强调步骤顺序，不要抽象化表达
- 必须指出关键节点和责任分工
- 必须包含异常处理或例外路径
- 输出要适合拆成分步 slide，而不是长段落说明
- 如果企业流程资料不完整，要明确标注“需企业补充流程节点”

### 2.4 系统操作模板

#### 适用场景

- ERP 系统培训
- CRM 系统培训
- OA 系统培训
- 内部业务系统上线培训
- 工具使用规范培训

#### 模板目标

帮助学员掌握系统操作的使用场景、关键入口、标准操作步骤、常见报错和错误操作规避方式。

#### 推荐输出结构

1. 系统用途与适用角色
2. 操作前准备
3. 关键入口与页面说明
4. 标准操作步骤
5. 常见报错 / 常见问题
6. 注意事项与错误示例
7. 上机练习 / 操作任务

#### Prompt 约束重点

- 输出必须分步骤，不要只讲概念
- 如果缺少真实页面资料，不要虚构具体按钮文案
- 应尽量使用“页面/入口/字段/提交结果”这种表达结构
- 每个关键步骤最好附带一个常见错误提醒
- 适合和截图、录屏、演示页搭配使用

### 2.5 技能模板

#### 适用场景

- 沟通技能培训
- 销售技能培训
- 管理技能培训
- 复盘与辅导培训
- 新技能 / 新工具使用能力培训

#### 模板目标

帮助学员从“知道概念”走向“会在岗位中使用”，强调业务场景、方法框架、示例、练习与迁移应用。

#### 推荐输出结构

1. 业务背景与痛点
2. 学习目标
3. 方法框架 / 能力模型
4. 步骤拆解
5. 正反案例
6. 常见误区
7. 练习设计
8. 行动建议 / 岗位迁移建议

#### Prompt 约束重点

- 必须从业务场景切入，而不是纯概念定义
- 至少提供一个真实工作场景或角色案例
- 必须给出常见误区与改进建议
- 必须体现“学完后怎么用”
- 输出语言要实战、简洁，避免学校课堂腔

### 2.6 产品 / 服务知识模板

#### 适用场景

- 产品知识培训
- 服务标准培训
- 方案介绍培训
- 客户沟通知识培训
- 售前 / 售后知识培训

#### 模板目标

帮助学员快速建立对产品、服务或方案的核心认知，并能在业务场景中准确表达、正确应用和处理常见问答。

#### 推荐输出结构

1. 培训目标与适用对象
2. 核心知识点总览
3. 重点概念与业务价值
4. 应用场景说明
5. 常见客户问题 / 常见业务问答
6. 容易混淆的点
7. 测验与记忆强化

#### Prompt 约束重点

- 重点突出产品价值与业务应用，不只是罗列参数
- 必须区分核心知识与辅助知识
- 应包含高频问答或业务交流场景
- 避免信息过载，适合拆分为多个知识页
- 对缺少企业资料的部分应保留占位或通用表述

### 2.7 安全 / 应急模板

#### 适用场景

- 消防安全培训
- 生产安全规程培训
- 食品安全培训
- 应急预案培训
- 职业健康与劳动保护培训
- 化学品/危险品操作安全培训

#### 模板目标

帮助学员掌握在高风险操作或紧急情况下的正确行为规范，强调操作前预防、应急时响应、事后复盘的完整逻辑，而不只是制度背诵。

#### 推荐输出结构

1. 培训背景与适用场景
2. 核心风险识别
3. 标准操作规程（SOP）
4. 紧急情况处置流程
5. 事故案例分析
6. 常见违规行为与后果
7. 应急演练设计
8. 法规要求与企业责任说明

#### Prompt 约束重点

- 必须区分常规制度类与安全操作类，将制度背诵与操作规程分开设计
- 高风险场景必须包含应急响应步骤，不只是规则列举
- 不得声称企业具体法规条款，对缺少企业资料的内容必须标注“需企业确认”
- 必须包含事故案例或违规后果，强化风险认知
- 语言要准确、动作导向，适合现场操作场景

#### 归类边界说明

当培训重点是“风险识别、现场动作、处置步骤、演练流程、事故响应”时，应优先归到安全 / 应急模板。

如果主题同时包含制度与应急，应优先按**主要训练目标**归类，而不是按字面名词归类。

## 3. 模板选择器设计

### 3.1 为什么需要模板选择器

模板库本身并不能解决泛化问题，真正关键的是在生成前先判断"当前需求更适合哪一类模板族"。

因此，产品层应增加一个模板选择器，用来在生成前完成以下判断：

1. 当前属于哪种培训类型
2. 当前是否有企业资料约束
3. 当前内容风险等级高不高
4. 当前更适合什么交付方式

### 3.2 选择器核心判断维度

建议至少判断四个维度：

- `trainingType`：onboarding / policy / process / system / skill / product / safety
- `sourceMode`：strict_grounded / grounded / template
- `riskLevel`：high / medium / low
- `deliveryMode`：online / offline / hybrid

### 3.3 选择逻辑示例

#### 示例 1：新员工考勤制度培训

建议识别为：

- `trainingType = policy`
- `sourceMode = strict_grounded`
- `riskLevel = high`
- `deliveryMode = online | offline`

推荐模板族：制度 / 合规模板

#### 示例 2：新员工入职总览培训

建议识别为：

- `trainingType = onboarding`
- `sourceMode = grounded | template`
- `riskLevel = medium`
- `deliveryMode = hybrid`

推荐模板族：入职 / 融入模板

#### 示例 3：CRM 系统操作培训

建议识别为：

- `trainingType = system`
- `sourceMode = grounded`
- `riskLevel = medium`
- `deliveryMode = online | hybrid`

推荐模板族：系统操作模板

#### 示例 4：销售经理复盘与辅导技能培训

建议识别为：

- `trainingType = skill`
- `sourceMode = grounded | template`
- `riskLevel = medium`
- `deliveryMode = workshop | hybrid`

推荐模板族：技能模板

#### 示例 5：新品知识培训

建议识别为：

- `trainingType = product`
- `sourceMode = grounded | template`
- `riskLevel = low | medium`
- `deliveryMode = online | offline`

推荐模板族：产品 / 服务知识模板

#### 示例 6：消防安全与应急疏散培训

建议识别为：

- `trainingType = safety`
- `sourceMode = strict_grounded`
- `riskLevel = high`
- `deliveryMode = offline | hybrid`

推荐模板族：安全 / 应急模板

#### 用户输入模糊时的处理策略

当用户输入过于宽泛（如“员工培训”、“新人入职”）时，系统不应直接猜测模板族，而应进行主动澄清追问。

建议追问策略：

1. 先展示七大模板族备选，让用户选择训练类型
2. 当选择不确定时，追问一个关键问题：“这次培训的主要目的是？”
3. 根据用户回答自动匹配最相符的模板族
4. 展示推荐结果并说明推荐理由，让用户确认或手动修改

### 3.4 选择器输出建议

模板选择器不应只输出“模板名称”，而应输出一个标准化结果对象，供后续所有 prompt 共用。

建议输出结构：

```ts
{
  trainingType: 'onboarding' | 'policy' | 'process' | 'system' | 'skill' | 'product' | 'safety',
  sourceMode: 'strict_grounded' | 'grounded' | 'template',
  riskLevel: 'high' | 'medium' | 'low',
  deliveryMode: 'online' | 'offline' | 'hybrid',
  templateFamily: string,
  assessmentNeeded: boolean,
  reviewPolicy: 'strict' | 'standard' | 'light',
}
```

其中：

- `templateFamily` 用于选择模板族
- `reviewPolicy` 用于决定后续 reviewer 的严格程度
- `assessmentNeeded` 用于决定是否强制生成测验、判断题或练习题

为支持“自动识别 + 必要时补问”的交互模式，建议在工程实现中进一步补充两类元信息：

- 字段来源：如 `user_input`、`inferred_from_text`、`inferred_from_docs`、`confirmed_by_dialog`
- 置信度：如 `high`、`medium`、`low`

这些元信息不一定要在第一版全部暴露给用户，但应作为系统内部判断依据，用于决定是否直接生成、是否进入补问、是否需要用户确认。

## 4. 模板输入与输出结构

### 4.1 模板输入字段设计

#### 设计目标

模板输入字段的作用，不是收集越多信息越好，而是让系统在信息不完整的情况下也能稳定生成课件，并且知道哪些地方必须谨慎、哪些地方可以通用化处理。

这里需要特别澄清：本节描述的主要是**系统内部目标字段模型**，不等于前端必须向用户逐项采集的大表单。

建议分成两层理解：

- 用户输入层：尽量保持一句话 requirement、附件上传、少量必要补问
- 系统策略层：内部派生 `trainingType`、`sourceMode`、`riskLevel`、`templateFamily`、`assessmentNeeded` 等结构化字段

因此，这里的字段设计更适合作为生成链路中的中间策略模型，而不是直接照搬为前端交互表单。

#### 建议的统一输入结构

建议所有模板族共享一套统一字段：

```ts
{
  trainingType: string,
  sourceMode: 'strict_grounded' | 'grounded' | 'template',
  audience: string,
  businessContext?: string,
  trainingGoal?: string,
  duration?: string,
  deliveryMode?: 'online' | 'offline' | 'hybrid',
  assessmentNeeded?: boolean,
  mustInclude?: string[],
  forbidden?: string[],
  tone?: 'formal' | 'professional' | 'practical' | 'friendly',
  riskLevel?: 'high' | 'medium' | 'low',
  sourceDocuments?: string[],
}
```

#### 核心字段说明

#### `trainingType`

用于决定走哪一类模板族，是第一优先级字段。

#### `sourceMode`

建议分三档：

- `strict_grounded`：必须严格依据企业资料，不可编造
- `grounded`：优先依据资料，可适度补足组织表达
- `template`：没有明确企业资料，输出通用模板与占位内容

**混合资料场景的动态判断规则：**

企业实际上传资料时，资料完整度往往参差不齐，建议按以下规则动态判断 sourceMode：

| 资料状态 | 推荐 sourceMode |
|---|---|
| 上传了完整制度/合规文件 | `strict_grounded` |
| 上传了部分资料 + 有口头描述 | `grounded` |
| 仅有口头描述或简单说明 | `grounded` |
| 完全没有企业资料 | `template` |
| 制度/合规类且资料不完整 | 强制 `strict_grounded`，缺失部分标注“需企业确认” |

> 注意：当用户上传资料后，系统应重新评估 sourceMode，不可在会话开始时一次性锁定。

#### `audience`

必须明确培训对象，例如：

- 新员工
- 一线员工
- 主管
- 销售经理
- 门店店长
- 区域经理

#### `mustInclude`

用于表达企业强要求，例如：

- 必须包含请假流程
- 必须包含违规后果
- 必须包含实操演练
- 必须包含判断题

#### `forbidden`

用于约束不要生成的内容，例如：

- 不要出现企业未确认的制度细则
- 不要输出过多理论解释
- 不要出现学校课堂式表达

#### 不同模板族的补充字段

虽然建议统一字段，但不同模板族仍可增加少量特有字段：

- 制度类：`complianceLevel`、`policyOwner`
- 流程类：`processOwner`、`exceptionRequired`
- 系统类：`systemName`、`rolesInSystem`
- 技能类：`scenarios`、`commonMistakesRequired`
- 产品类：`productLine`、`faqRequired`



### 4.2 模板输出结构设计

#### 设计原则

模板的输出结构不应只是“生成一段看起来专业的说明”，而应直接服务于课件生产。

因此输出结构建议满足四个目标：

1. 便于生成课程大纲
2. 便于生成页级课件内容
3. 便于生成讲师备注
4. 便于生成测验、案例与互动环节

#### 推荐的标准输出层级

建议模板输出至少分为四层：

#### 第一层：课程定义层

用于明确课程边界，建议输出：

- 课程名称
- 适用对象
- 培训目标
- 业务问题或培训背景
- 课程价值主张
- 培训边界与不包含内容

#### 第二层：课程模块层

用于形成大纲，建议输出：

- 模块名称
- 模块目标
- 模块关键知识点
- 模块建议时长
- 模块建议互动方式

#### 第三层：页级课件层

用于真正生成 slide 或 scene，建议输出：

- 页面标题
- 本页核心信息
- 3-5 个要点
- 示例 / 场景 / 图示建议
- 讲师提示语
- 是否需要 quiz / 讨论 / 案例

#### 第四层：评估与强化层

用于课后检验与巩固，建议输出：

- 判断题 / 选择题 / 场景题
- 高频误区回顾
- 行动建议
- 培训确认点

#### 不同模板族的输出差异

#### 入职 / 融入类

重点输出：

- 新人必须知道的内容
- 高频场景说明
- 风险提醒
- 常见问题与确认题

#### 制度 / 合规类

重点输出：

- 规则点
- 边界
- 例外
- 风险后果
- 场景判断题

#### 流程 / SOP 类

重点输出：

- 步骤
- 关键节点
- 责任角色
- 异常处理
- 常见返工点

#### 系统操作类

重点输出：

- 关键入口
- 标准步骤
- 页面说明
- 常见报错
- 操作练习

#### 技能类

重点输出：

- 痛点
- 方法框架
- 正反案例
- 常见误区
- 实战练习

#### 产品 / 服务知识类

重点输出：

- 核心知识点
- 业务价值
- 应用场景
- FAQ
- 测验与记忆点

## 5. 产品化落地建议

### 5.1 产品层不要只暴露“模板名”

用户不一定知道自己该选什么模板，因此产品层不应只暴露一个模板列表。

更合理的方式是：

1. 用户输入需求
2. 系统先识别 trainingType / sourceMode / riskLevel
3. 系统自动推荐模板族
4. 用户可接受推荐或手动切换

### 5.2 模板应贯穿全链路

模板体系不应只作用于某一个 prompt，而应贯穿整个生成链路：

- 需求识别 prompt
- agent profiles 生成
- director prompt
- 课程大纲生成
- 场景 / 页面生成
- quiz 生成
- reviewer / 自检 prompt

### 5.3 模板与企业资料必须联动

模板体系必须与资料模式联动：

- `strict_grounded`：强资料约束
- `grounded`：资料优先，允许教学化整理
- `template`：输出通用模板与占位项

尤其对制度类课件，资料联动是产品风险控制的核心。

### 5.4 模板体系要支持企业沉淀

真正做成产品后，企业会希望沉淀自己的培训资产。

因此建议支持：

- 企业保存自己的模板
- 企业保存自己的固定字段
- 企业保存自己的固定讲师备注格式
- 企业保存自己的页面结构偏好
- 企业保存自己的题型偏好

### 6.1 最适合接入模板体系的模块

结合当前代码结构，模板体系最适合接入以下位置：

#### 2）Agent 人设与编排层

- `lib/orchestration/registry/store.ts`

- `lib/orchestration/director-prompt.ts`

这里适合增加：

#### 3）场景与页面生成层

- `lib/generation/prompts/*`
- `lib/generation/scene-generator.ts`
- `lib/generation/scene-builder.ts`

这里适合增加：

- 不同模板族对应的场景结构要求
- 不同模板族对应的页级输出规则
- 不同模板族对应的 quiz / case / practice 生成策略

### 6.2 推荐的数据流

建议未来形成如下数据流：

1. 用户输入培训需求
2. 系统识别 `trainingType/sourceMode/riskLevel`
3. 模板选择器产出 `templateSelectionResult`
4. 该结果传递给：
   - agent profiles 生成
   - director prompt
   - outline prompt
   - scene generation prompt
   - reviewer prompt
5. 所有生成结果统一受同一模板配置约束

### 6.3 推荐新增的数据对象

建议新增一个统一模板选择结果对象：

```ts
interface TemplateSelectionResult {
  trainingType: 'onboarding' | 'policy' | 'process' | 'system' | 'skill' | 'product' | 'safety';
  sourceMode: 'strict_grounded' | 'grounded' | 'template';
  riskLevel: 'high' | 'medium' | 'low';
  deliveryMode: 'online' | 'offline' | 'hybrid';
  templateFamily: string;
  assessmentNeeded: boolean;
  reviewPolicy: 'strict' | 'standard' | 'light';
  mustInclude?: string[];
  forbidden?: string[];
}
```

这样可以避免每个 prompt 各自推断培训类型，降低系统漂移。

如果后续采用“自动识别 + 必要时补问”的主交互，还建议为该对象补充两类辅助信息：

- 字段来源：用于区分来自用户明确输入、文本推断、附件推断还是补问确认
- 置信度：用于决定是否直接生成、是否进入补问、是否需要用户确认

换句话说，`TemplateSelectionResult` 不只是结论对象，也应逐步演进为生成链路中的策略判断对象。

## 7. Prompt 改造建议

### 7.1 改造优先级建议

如果要在现有代码上继续优化，建议按以下顺序推进：

#### P0：引入 `trainingType`

这是最优先变量。

没有 `trainingType`，系统很难稳定区分：

- 制度类课件
- 入职类课件
- 流程类课件
- 技能类课件
- 系统操作类课件

#### P1：引入 `sourceMode`

这是企业产品风险控制核心变量。

建议强制区分：

- `strict_grounded`：制度/合规高风险内容
- `grounded`：有资料但允许教学化整理
- `template`：没有明确资料，只能生成通用模板

#### P2：引入固定输出结构

至少要求输出包含：

- 课程定义
- 模块结构
- 页级设计
- 测验/练习建议
- 讲师备注建议


#### P3：引入 Reviewer / 自检层

用于在生成后增加一轮企业培训视角审查，重点检查：

- 是否像企业培训而不是学校课堂
- 是否存在空泛表达
- 是否缺少场景、练习、测验或边界说明
- 是否存在制度类幻觉或越权编造
- 是否一页信息过载、不适合实际培训

### 7.2 结合现有模块的改造建议

#### `app/api/generate/agent-profiles/route.ts`

建议增加：

- 输入 `trainingType`
- 输入 `sourceMode`
- 按模板族生成不同 Agent 组合
- 对制度类场景强调严谨与边界
- 对技能类场景强调业务场景、练习与迁移

#### `lib/orchestration/prompt-builder.ts`

建议增加：

- `Template Focus` 段落
- 按模板族切换发言重点
- 按 `sourceMode` 切换是否允许自由补充
- 强制 Agent 每轮至少补一个课件要素：规则点、案例、练习、风险、FAQ、讲师提示语等

#### `lib/orchestration/director-prompt.ts`

建议增加：

- 按模板族判断当前缺口
- 优先调度能补齐缺失要素的 Agent
- 对制度类优先补规则/边界/后果
- 对技能类优先补场景/方法/误区/练习

#### `lib/generation/outline-generator.ts`

建议增加：

- 根据模板族生成不同的大纲结构
- 根据 `assessmentNeeded` 决定是否强制增加测验页
- 根据 `riskLevel` 决定是否增加“确认页 / 风险提醒页”

#### `lib/generation/prompts/*`

建议增加：

- 不同模板族的场景生成 prompt 片段
- 不同模板族的页级结构约束
- 不同模板族的练习与测验生成规则

#### Prompt 片段与配置中心的职责划分

为了避免后续规则散落在多个 prompt 文件中，建议从一开始就明确：

- prompt snippet 负责内容：文案规则、表达约束、输出重点、reviewer 检查逻辑
- 配置中心负责映射：模板族到 snippet 的映射、默认 riskLevel、允许的 sourceMode、默认 output focus、assessmentNeeded、reviewer 类型

可以把它简单理解为：**prompt snippet 是内容，配置中心是路由表。**

## 8. 实施路线与任务清单

### 8.1 建设路线

#### Phase 1：系统模板族 + 轻量兜底机制上线

上线 7 个系统模板族：

- 入职 / 融入
- 制度 / 合规
- 流程 / SOP
- 系统操作
- 技能
- 产品 / 服务知识
- 安全 / 应急

本阶段的主目标，不是把产品做成“手动选模板页面”，而是先打通最小可用闭环：

- 一句话输入培训需求
- 系统做最小自动识别
- 信息不足时进行轻量补问
- 在识别不稳定时，允许通过轻量选择器兜底纠偏

如果 Phase 1 需要提供 trainingType 下拉选择，它更适合作为过渡方案、灰度方案或内部纠偏入口，而不是长期主交互。

> 说明：Phase 1 不代表“以手动选择为主”，而是在自动识别尚未稳定前，允许轻量兜底干预，避免用户因选错模板而影响体验。

目标：先覆盖多数常见企业培训场景，同时建立“一句话输入 → 自动识别 / 必要时补问 → 生成”的最小闭环。

#### Phase 2：模板选择器智能化

在 Phase 1 的基础上，进一步增强自动识别能力：

- 基于用户输入自动推断 trainingType
- 基于上传附件自动判断 sourceMode
- 支持意图模糊时的自动追问流程
- riskLevel / deliveryMode 自动识别与推荐
- 尽量减少用户手动干预频率

目标：让模板使用从“有兜底的识别流程”升级为“自动识别优先的推荐流程”。

#### Phase 3：企业自定义模板中心

增加能力：

- 企业创建自定义模板
- 企业保存固定字段默认值
- 企业保存固定页面结构偏好
- 企业保存固定讲师备注格式
- 企业保存固定题型偏好
- 企业保存禁用表达与品牌术语

目标：让系统模板升级为“企业级模板资产”。

#### Phase 4：模板全链路接入

将模板体系接入整个生成流程：

- 需求识别
- 模板选择
- Agent Profiles 生成
- Director Prompt
- 大纲生成
- 场景生成
- Quiz 生成
- Reviewer 自检

目标：让模板不只影响一个 prompt，而是影响整条生产链。

#### Phase 5：模板效果评估与优化

增加模板效果评估机制：

- 不同模板族的生成稳定性分析
- 不同行业企业的适配反馈
- 用户手动切换模板的数据回流
- Reviewer 拒绝/修正原因统计
- 课件生成后人工修改热点分析

目标：让模板体系从“可用”逐步升级到“可优化、可演进”。

## 10. 面向工程执行的 Backlog 化 Implementation Plan

### 10.1 目标与落地原则

本章节将前文的目标设计进一步翻译为面向研发执行的 backlog 形式，重点聚焦三条主线：

1. 模板选择器（Template Selector）
2. 统一策略对象（TemplateSelectionResult / TrainingStrategy）
3. Prompt 分层治理（识别层 + 路由层 + snippet + reviewer）

本 implementation plan 默认遵循以下工程落地原则：

- 不改变当前一句话输入为主的核心交互
- 优先做生成前 preflight 识别，而不是先做复杂模板管理 UI
- 优先建立统一策略对象，再做多模块联动改造
- 高风险模板族优先做风险控制，尤其是制度 / 合规类
- Prompt 治理采用 phased rollout，而不是一次性全量重写

### 10.2 总体研发拆解

建议按五个 Phase 推进。

#### Phase 0：策略模型与枚举口径收敛

目标：先统一概念、字段、模板边界和默认规则，避免后续实现口径漂移。

核心输出：

- 七大模板族边界定义
- `trainingType` / `templateFamily` / `sourceMode` / `riskLevel` / `deliveryMode` 枚举口径
- 与当前 `UserRequirements.requirement` 的兼容映射
- `TemplateSelectionResult` 初版 schema
- 高风险模板族默认策略表

#### Phase 1：模板选择器最小闭环

目标：在不改变一句话输入主交互的前提下，打通“输入 → preflight 识别 → 必要时补问 / 轻量选择器 → 生成”的最小闭环。

核心输出：

- 生成入口 preflight 模板识别
- 低置信度补问机制
- 轻量选择器兜底机制
- `sourceMode` 动态重评估能力
- 模板识别摘要卡 / 内部调试信息

#### Phase 2：统一策略对象贯通生成链路

目标：让模板识别结果成为整条生成链路共享的单一事实来源。

核心输出：

- `TemplateSelectionResult` / `TrainingStrategy` 类型落地
- API payload、orchestration context、generation metadata、review input 统一透传
- outline / director / scene / quiz / reviewer 统一读取策略对象
- 减少各模块重复推断培训类型和风险等级

#### Phase 3：Prompt 分层治理与配置中心

目标：把模板族差异管理从“散落在多个 prompt 中”升级为“结构化配置 + snippet + reviewer policy”的治理体系。

核心输出：

- 识别层 prompt
- 路由层配置中心
- 模板族 snippet registry
- reviewer policy 分层
- outline / director / reviewer 优先接入的 phased rollout

#### Phase 4：评估回流与灰度优化

目标：建立模板治理的评估闭环，而不是一次性静态上线。

核心输出：

- 识别命中率、手动切换率、补问触发率埋点
- reviewer 拒绝原因统计
- 模板族稳定性评估
- 人工修改热点分析
- 基于数据的 prompt / 路由 / reviewer 持续优化

### 10.3 Phase 0：策略模型与枚举口径收敛

#### 目标

统一所有核心字段的枚举、定义和边界，为后续实现提供单一基线。

#### 工程任务

- [ ] 明确七大模板族边界，以“主要训练目标”作为归类优先原则
- [ ] 统一 `trainingType` 与 `templateFamily` 的关系，第一版允许同值但保留独立字段
- [ ] 统一 `sourceMode` 三档定义：`strict_grounded` / `grounded` / `template`
- [ ] 明确 `riskLevel` 默认映射：policy / compliance 高风险优先
- [ ] 统一 `deliveryMode`、`assessmentNeeded`、`mustInclude`、`forbidden` 字段语义
- [ ] 明确当前 `UserRequirements.requirement` 到结构化策略对象的兼容派生逻辑
- [ ] 输出 decision doc，作为后续 prompt、类型和 API 改造依据

#### 验收标准

- 七大模板族边界不再出现明显歧义
- `sourceMode` 与高风险模板族的组合规则明确
- requirement-only 输入可以稳定映射为内部策略字段
- 后续各模块不再自行定义不同版本的字段口径

### 10.4 Phase 1：模板选择器最小闭环

#### 目标

让系统在生成前具备模板识别能力，并在识别不稳定时通过补问或轻量选择器完成兜底。

#### 子能力 A：模板选择器 I/O

输入建议包括：

- requirement 自由文本
- 附件 / 企业资料状态
- 会话中的补问信息
- 用户是否手动切换模板族

输出建议包括：

- 推荐 `trainingType`
- 推荐 `templateFamily`
- 推荐 `sourceMode`
- 推荐 `riskLevel`
- 是否需要补问
- 是否展示轻量选择器
- 字段来源与置信度

#### 子能力 B：补问与轻量选择器

建议采用三档置信度：

- `high`：直接生成
- `medium`：直接生成，但展示识别摘要并允许轻量切换
- `low`：触发补问，必要时使用轻量选择器兜底

补问只应聚焦最关键差异，例如：

- 更偏规则说明还是步骤操作？
- 是否有企业制度 / SOP / 系统页面资料？
- 是否需要测验 / 判断题 / 练习？
- 培训对象是谁？

#### 子能力 C：sourceMode 动态重评估

以下事件发生时应重新评估 `sourceMode`：

- 用户上传资料
- 用户补充业务描述
- 用户切换模板族
- reviewer 判定资料与输出不匹配

制度 / 合规类在资料不完整时，应强制保守策略：

- 优先按 `strict_grounded` 处理
- 缺失内容标记“需企业确认”

#### 工程任务

- [ ] 在生成入口之前增加 preflight 模板识别阶段
- [ ] 新增低置信度补问策略
- [ ] 新增轻量选择器兜底逻辑
- [ ] 实现 `sourceMode` 动态重评估
- [ ] 将模板识别结果以摘要形式回传前端或记录调试日志

#### 验收标准

- 常见企业培训需求能稳定归类到 7 大模板族
- 识别不稳定时系统能补问或给出轻量切换入口
- 不需要用户先填写复杂表单才能进入生成
- 高风险场景不会因为缺资料而直接走宽松生成

### 10.5 Phase 2：统一策略对象贯通生成链路

#### 目标

让模板识别结果从“入口阶段的判断结果”升级为“生成链路共享的单一事实来源”。

#### 关键类型设计

建议在现有需求结构基础上增加统一策略对象，例如：

- `trainingType`
- `templateFamily`
- `sourceMode`
- `riskLevel`
- `deliveryMode`
- `assessmentNeeded`
- `mustInclude`
- `forbidden`
- `fieldSources`
- `confidence`
- `clarificationNeeded`

其中：

- `trainingType` 用于表达训练目标
- `templateFamily` 用于路由模板族 snippet 与 reviewer policy
- `sourceMode` 用于控制资料约束强度
- `riskLevel` 用于控制 reviewer 严格度与输出风险策略

#### 模块级 Backlog

##### A. 类型层（Types）

- [ ] 在 `lib/types/generation.ts` 增加 `TemplateSelectionResult` 或等价策略对象
- [ ] 为 `confidence`、`fieldSources`、`assessmentNeeded` 等字段定义明确类型
- [ ] 保留与旧版 `UserRequirements` 的兼容层，确保 requirement-only 输入仍可工作
- [ ] 为高风险模板族定义保守默认值

##### B. API / Payload 层

- [ ] 在生成入口 payload 中加入 preflight 识别结果
- [ ] 支持用户手动确认后的模板族覆盖值
- [ ] 支持记录当前资料状态，供 `sourceMode` 重评估使用
- [ ] 为后续埋点保留 `templateFamily`、`sourceMode`、`confidence` 元信息

##### C. Orchestration Context 层

- [ ] 将统一策略对象挂入 orchestration context，供下游模块读取
- [ ] 避免 outline、director、scene、quiz、reviewer 各自重复推断培训类型
- [ ] 确保策略对象在多阶段生成中不会丢失或被局部重写

##### D. Generation / Metadata 层

- [ ] 在最终课件 metadata 中记录本次生成使用的 `templateFamily`
- [ ] 记录 `sourceMode`、`riskLevel`、`assessmentNeeded`、`confidence`
- [ ] 为后续问题回溯提供策略快照

##### E. Reviewer Input 层

- [ ] reviewer 从统一策略对象读取模板族与风险等级
- [ ] reviewer 不再重新猜测需求类型，而是只做一致性校验
- [ ] 高风险模板族使用更严格的 reviewer policy

#### 验收标准

- 各生成模块消费的是同一份策略对象，而不是各自局部推断
- metadata 能回溯本次生成所采用的模板与风险策略
- `strict_grounded` 场景下 reviewer 能识别并拒绝明显编造内容
- 技能 / 流程 / 系统类内容能体现模板族差异，而不是输出同质化课件

### 10.6 Phase 3：Prompt 分层治理与配置中心

#### 目标

将模板族差异从“散落在 prompt 文本中的隐式逻辑”提升为“结构化、可配置、可灰度”的治理体系。

#### Prompt 分层建议

##### A. 识别层 Prompt

职责：

- 根据 requirement、资料状态、补问信息输出结构化策略对象
- 输出字段值、字段来源和置信度
- 不直接承担课件正文生成责任

##### B. 路由层

职责：

- 根据 `templateFamily` 决定调用哪个 snippet
- 决定默认 `reviewPolicy`
- 决定允许的 `sourceMode`
- 决定输出重点与默认 `assessmentNeeded`

##### C. 模板族 Snippet 层

职责：

- 表达各模板族的生成重点、表达约束和结构偏好

示例重点：

- onboarding：新人友好、分层清晰、帮助快速融入
- policy：规则、边界、例外、后果、需确认项
- process：步骤、角色、节点、异常处理
- system：页面入口、操作步骤、结果反馈、常见报错
- skill：场景、误区、练习、迁移
- product：业务应用、常见问题、卖点与边界
- safety：风险识别、后果、防护、应急处理

##### D. Reviewer 层

职责：

- 按模板族检查输出重点是否命中
- 按 `riskLevel` 检查是否存在高风险幻觉
- 按 `sourceMode` 检查是否违反资料约束

##### E. 配置中心

建议维护：

- `templateFamily -> snippetKey`
- `templateFamily -> reviewPolicy`
- `templateFamily -> allowedSourceModes`
- `templateFamily -> defaultRiskLevel`
- `templateFamily -> outputFocus`
- `templateFamily -> defaultAssessmentNeeded`

#### 模块级 Backlog

##### A. Prompt Builder / Registry

- [ ] 将模板族差异从内联 prompt 中抽离为 snippet registry
- [ ] 支持按 `templateFamily` 路由不同 prompt 片段
- [ ] 支持 reviewer policy 与 snippet 映射表

##### B. 高价值模块优先接入

- [ ] 优先改造 outline prompt
- [ ] 优先改造 director prompt
- [ ] 优先改造 reviewer prompt
- [ ] 第二阶段扩展到 agent profiles、scene、quiz

##### C. 治理与灰度

- [ ] 支持按模板族灰度启用新 snippet
- [ ] 支持按 reviewer policy 回滚
- [ ] 支持用配置开关控制 phased rollout

#### 验收标准

- 模板族差异可通过配置与 snippet 稳定表达
- reviewer 能根据模板族执行差异化校验逻辑
- 不需要在多个 prompt 中重复维护相同模板规则
- outline / director / reviewer 三处输出能体现一致的模板策略

### 10.7 Phase 4：评估回流与灰度优化

#### 目标

让模板体系具备可观测性、可评估性和可持续优化能力。

#### 观测指标建议

##### A. 识别层指标

- 自动识别命中率
- 手动切换率
- 低置信度占比
- 补问触发率
- 不同模板族的识别分布

##### B. 生成层指标

- reviewer 拒绝率
- reviewer 拒绝原因分布
- 模板模式下占位项完整率
- 高风险模板族误生成率

##### C. 业务反馈指标

- 用户接受推荐模板比例
- 再生成率
- 人工修改频次
- 人工修改热点模块

#### 工程任务

- [ ] 在生成记录中增加模板族与置信度埋点
- [ ] 统计手动切换、补问、重评估触发情况
- [ ] 记录 reviewer 拒绝原因并做模板族聚合分析
- [ ] 建立模板测试集与定期回归机制
- [ ] 根据埋点结果持续优化 prompt、路由和 reviewer policy

#### 验收标准

- 可以区分识别问题、资料问题、prompt 问题和 reviewer 问题
- 可以看出哪些模板族最不稳定、最依赖人工修订
- 可以支撑灰度发布、回滚和版本对比
- 模板优化不再依赖零散主观反馈，而有数据依据

### 10.8 推荐模块映射与实施顺序

建议按“闭环优先、最小侵入、逐步扩散”的方式落地。

#### 第一批：必须改造

- 生成入口：增加 preflight 识别
- 类型定义：增加统一策略对象
- orchestration context：透传策略对象
- outline：读取模板策略
- director：读取模板策略
- reviewer：按模板族审查

#### 第二批：逐步扩展

- scene 生成
- quiz 生成
- agent profiles
- generation metadata 展示与调试面板

#### 第三批：治理与资产化

- 模板族配置中心
- 企业自定义模板映射能力
- 模板效果评估看板

### 10.9 接口 / 类型 / Prompt 变更清单

#### 接口变更

- [ ] 在生成入口增加 preflight 识别调用
- [ ] 在生成 API payload 中增加模板策略字段
- [ ] 支持前端传入用户手动确认或覆盖结果

#### 类型变更

- [ ] 扩展 `UserRequirements` 或并行新增 `TemplateSelectionResult`
- [ ] 增加 `confidence` / `fieldSources` / `clarificationNeeded` 等辅助字段
- [ ] 为 metadata 增加模板策略快照字段

#### Prompt 变更

- [ ] 增加识别层 prompt
- [ ] 增加模板族 snippet registry
- [ ] 为 outline / director / reviewer 接入模板族路由
- [ ] 按高风险模板族强化 reviewer 提示词

#### 测试变更

- [ ] 为七大模板族建立代表 case 测试集

- [ ] 为轻量选择器和补问逻辑建立交互测试
- [ ] 为 `sourceMode` 重评估建立回归测试

### 10.10 最终研发建议

如果只能优先做一小步，建议优先级如下：

1. 先落 `TemplateSelectionResult` 类型和 preflight 识别
2. 再把策略对象透传到 outline / director / reviewer
3. 最后做 prompt 分层治理和配置中心

原因是：

- 没有 preflight，系统无法稳定决定“该按什么模板生成”
- 没有统一策略对象，后续模块会继续各自推断、相互漂移
- 没有 prompt 分层治理，模板族规则将持续散落、难以维护

因此本计划的工程核心不是“多加几个模板”，而是建设：

- 生成前的模板识别层
- 生成中的统一策略层
- 生成后的 reviewer 与评估治理层

这三层能力打通之后，模板体系才会真正成为企业培训产品的基础设施，而不是临时 prompt patch。


## 9. 最终结论

企业培训产品不适合预制大量“具体主题内容模板”，但非常适合预制“生成逻辑模板”。

最可行的路线不是做一个万能模板，也不是做无数细碎模板，而是：

- 用少量模板族覆盖大多数培训场景
- 用模板选择器解决泛化问题
- 用 `sourceMode` 解决企业资料约束问题
- 用统一输出结构保证课件可生产、可复审、可落地
- 用企业自定义模板沉淀长期资产

一句话总结：

> 预制模板可行，但应预制的是企业培训课件的生成逻辑，而不是具体培训内容本身。


### 9.1 最终建议

如果将 OpenMAIC 包装为企业培训产品，模板体系最合理的定位不是“预制具体培训课件”，而是“预制企业培训课件生成逻辑”。

因此建议产品策略定为：

1. 用 7 个系统模板族覆盖多数常见培训类型
2. 用模板选择器解决训练主题不可预期的问题
3. 用 `sourceMode` 控制制度类、合规类的资料约束风险
4. 用统一输入字段和统一输出结构提升生成稳定性
5. 用企业自定义模板沉淀客户长期资产

### 9.2 对研发落地的直接含义

这意味着后续研发不应只继续微调单个 prompt，而应逐步建设以下能力：

- 需求识别层：先判断培训类型与资料模式
- 模板选择层：将用户需求映射到模板族
- Prompt 编排层：所有 prompt 都读取同一模板选择结果
- Reviewer 层：按模板族进行质量检查
- 企业资产层：支持客户保存自己的模板配置

### 9.3 文档结尾结论

企业培训产品的泛化能力，不来自“一个万能 prompt”，而来自“模板族 + 选择器 + 资料约束 + 统一输出结构 + 企业自定义模板”的组合能力。

这也是当前项目从“课堂生成系统”走向“企业培训课件生产平台”最合适的演进方向。
